## v1.1 Requirements — Shipped 2026-05-21

### Auto-Strip Hook

- [x] **HOOK-01**: TUI main menu shows a third option **"Add co-author auto-strip hook"** alongside the existing rename and drop options
  *Outcome: `MenuChoice::all()` returns `[Rename, Drop, AddHook, ManageHook]`; `% 4` modulus navigation. `test_main_menu_shows_four_options` passes. Shipped in Phase 6 plan 06-02.*

- [x] **HOOK-02**: TUI main menu shows a fourth option **"Manage auto-strip hook"** alongside the other three options; this option is always visible (works even when no hook is installed, showing an empty state)
  *Outcome: ManageHook branch with `Absent` → `Screen::HookSuccess { state: Absent }`; render emits "No hook installed — no emails configured.". `test_main_menu_routes_manage_hook_empty` passes. Shipped in Phase 6 plan 06-02/06-04.*

- [x] **HOOK-03**: When the user picks "Add", the tool shows the list of currently-stripped emails (or "no entries yet" if none), then presents a fuzzy-filterable co-author list reusing the same enumeration as the existing drop flow
  *Outcome: `event.rs` AddHook branch calls `crate::git::reader::enumerate_coauthors` (same function as Drop branch; no parallel implementation). Shipped in Phase 6 plan 06-03.*

- [x] **HOOK-04**: When the user selects a co-author to add, the tool writes `.git/hooks/commit-msg` if absent, or rewrites it with the email appended to the embedded strip list if present and tool-managed
  *Outcome: `install_strip` in `mod.rs` handles fresh install and append-to-existing. `test_install_fresh_writes_file_with_markers_and_email`, `test_install_appends_to_existing_tool_managed_hook` pass. Shipped in Phase 5 plan 05-04.*

- [x] **HOOK-05**: Adding an email already present in the strip list is a no-op shown to the user as "already stripped: <email>" — the hook file is not rewritten
  *Outcome: `mod.rs` case-insensitive check returns `Ok(AddResult::AlreadyStripped)`; file bytes verified unchanged. `test_install_duplicate_email_is_noop_file_bytes_identical` passes. Shipped in Phase 5 plan 05-04.*

- [x] **HOOK-06**: If `.git/hooks/commit-msg` exists and does NOT contain the tool's marker comment, the tool refuses to overwrite, displays a clear error naming the file and instructing the user to remove or rename it, and exits the flow without writing anything
  *Outcome: `mod.rs` returns `Err(AppError::HookExists(p))`; `error.rs` message names the file and gives remediation instructions. `test_install_refuses_to_overwrite_non_tool_managed_hook` passes. Shipped in Phase 5 plan 05-04.*

- [x] **HOOK-07**: The generated hook file is POSIX `sh` (shebang `#!/bin/sh`, no bash-isms) and is created with executable permissions (mode 0755 on Unix); the strip list lives between two distinctive marker comments inside the file
  *Outcome: `render.rs` emits `#!/bin/sh\n`; `write.rs` sets `0o755` via `PermissionsExt`. `test_install_sets_mode_0755_on_unix`, `test_generated_hook_has_posix_shebang_and_markers` pass. Shipped in Phase 5 plans 05-03/05-04.*

- [x] **HOOK-08**: The hook strips lines from the commit message using the SAME case-insensitive `Co-authored-by:` matching semantics as the existing drop flow — trailing whitespace and trailer key case must match the parser's behavior
  *Outcome: `reader.rs` rfind+trim semantics twinned in `render.rs` backwards substr loop + gsub trim. Shell execution tests confirm parity. `test_shell_hook_strips_case_insensitive_matches`, `test_shell_hook_preserves_when_email_only_in_name_slot` pass. Shipped in Phase 5 plans 05-03/05-04.*

- [x] **HOOK-09**: When the user picks "Manage", the tool shows a fuzzy-filterable list of configured strip emails; selecting an entry removes it from the list and rewrites the hook file
  *Outcome: `HookManageList` with `build_strip_nucleo` + `apply_strip_filter`; three-zone layout in `render_hook_manage_list`. `test_manage_hook_list_shows_emails` passes. Shipped in Phase 6 plan 06-04.*

- [x] **HOOK-10**: Removing the last entry from the strip list deletes the hook file entirely — no empty marker file is left behind
  *Outcome: `remove_strip` calls `write::delete_hook` when the filtered list is empty; file verified absent. `test_remove_last_entry_deletes_file` asserts `HookDeleted` and `!hook_path.exists()`. Shipped in Phase 5 plan 05-04.*

- [x] **HOOK-11**: Both "Add" and "Manage" flows end on a success screen showing the resulting strip-list state (e.g. "Hook installed — stripping 2 emails: a@x.com, b@y.com" or "Hook removed — no entries remain")
  *Outcome: Post-install `read_strip_list` re-read at `event.rs` line 343; post-remove re-read at line 387. `HookDeleted` routes to `Screen::HookRemoved` (no re-read — file is gone). Engine is source of truth. Shipped in Phase 6 plans 06-03/06-04/06-06.*

- [x] **HOOK-12**: Hook install/manage operations do NOT trigger the existing stash/worktree pre-flight blockers (SAFE-01, SAFE-02) — installing a hook does not rewrite history
  *Outcome: `src/main.rs` has zero `check_stash`/`check_worktrees` calls; `check_stash` only at `event.rs` lines 48 (Rename) and 75 (Drop). `test_add_hook_no_preflight_with_stash`, `test_manage_no_preflight_with_stash` pass. Shipped in Phase 5 (engine) + Phase 6 plan 06-01 (structural fix).*

- [x] **HOOK-13**: Hook engine has automated Rust tests covering every code path: fresh install on a repo with no hook, append-to-existing tool-managed hook, no-op when adding a duplicate email, refuse-to-overwrite a non-tool-managed hook, remove a single entry, remove the last entry (hook file deleted), parse a tool-managed hook back into its strip list, mode 0755 is set, and the generated shell script correctly strips matching `Co-authored-by:` lines case-insensitively when executed against fixture commit messages
  *Outcome: 12 integration tests in `tests/hook_test.rs`; 9 unit tests in `parse.rs`, 14 in `render.rs`, 7 in `write.rs`. `cargo test` → 153 passed. Shipped in Phase 5 plan 05-05.*

- [x] **HOOK-14**: Hook TUI flows have automated tests covering every user path: main menu shows all four options and routes each correctly, "Add" happy path lands on success screen with updated list, "Add" with already-stripped email lands on no-op screen, "Manage" empty state renders when no hook installed, "Manage" remove-single-entry lands on success screen with remaining list, "Manage" remove-last-entry lands on "hook removed" screen, and neither flow invokes the SAFE-01/SAFE-02 preflight on a repo with stash entries
  *Outcome: 153 tests pass covering all TUI state machine paths (verified at 06-06 completion). Shipped in Phase 6 plans 06-02 through 06-06.*
